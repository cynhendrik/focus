Design — Phase 1: Store-Refactor + Filesystem

Das Fundament: Crash-Risiko eliminieren, Performance verbessern

Problem heute

Zustand: Ein einziger Zustand-Blob → localStorage (Key: cynera-os-v3)
Dateien: Base64-kodiert im selben Blob → bei ~3 PDFs ist das 5–10 MB → Browser verwirft den Key stillschweigend
Performance: Jeder State-Update → ganzer Blob wird serialisiert + gespeichert, inkl. aller Base64-Daten
Kein Memoizing: Gefilterte Listen (z.B. Todos eines Kunden) werden bei jedem Render neu berechnet

Lösung: Store in zwei Schichten aufteilen

ZUSTAND (bleibt in Zustand / localStorage)
customers[ ]
todos[ ]
notes[ ]
kpis[ ]
folders[ ]
uploadedFiles[ ] ← nur Metadaten
id, name, type, size
tauriPath ← neu
data (base64) entfernt
TAURI FILESYSTEM (neu)
AppData/cynera/
  files/
    {customerId}/
      {fileId}.{ext}
  workspace-files/
    {fileId}.{ext}
Lesen/Schreiben via Tauri fs API — kein localStorage-Limit

Store-Performance: Selektoren + Memoization

Heute: Jede Komponente ruft store.todos.filter(t => t.customerId === id) direkt auf
→ bei jedem Store-Update werden alle Komponenten neu gerendert
Neu: Selektoren als computed slices im Store (Zustand subscribeWithSelector)
+ useMemo in Komponenten für teure Filter/Sort-Operationen
+ Store in 3 Slices aufteilen: dataSlice · uiSlice · filesSlice

Was sich für den User ändert

✓ Dateien unbegrenzt hochladen (kein 5MB-Crash mehr)
✓ App spürbar schneller bei vielen Kunden/Todos
✓ Bestehende Dateien werden automatisch migriert (einmaliger Import beim ersten Start)
! Ablage-Download-Button bleibt gleich — liest jetzt von Disk statt aus Base64
