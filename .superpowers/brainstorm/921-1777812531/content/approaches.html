3 Umsetzungs-Ansätze

Alle führen zum Ziel — aber auf unterschiedlichen Wegen

A

Inkrementell — Phase für Phase

Jede Phase ist in sich abgeschlossen und liefert echten Mehrwert, bevor die nächste beginnt.

Phase 1 Store-Refactor + Tauri Filesystem (Dateien raus aus localStorage)
Phase 2 Dashboard + Markdown-Editor Upgrade (remark-gfm, Syntax-Highlighting)
Phase 3 Persönlicher Workspace in Sidebar (Notizen + Todos ohne Kundenbezug)
Phase 4 Rich-Editor (Slash-Commands + [[Backlinks]] + Volltextsuche)
✓ Empfohlen · Sicherste Route · Kein Rework · Crash-Risiko wird sofort eliminiert
B

Editor-First

Erst den Markdown-Editor zur vollwertigen Rich-Text-Engine ausbauen — dann baut der persönliche Workspace darauf auf.

Schritt 1 Rich-Editor (Tiptap/ProseMirror + Slash-Commands + Backlinks)
Schritt 2 Persönlicher Workspace (nutzt sofort den neuen Editor)
Schritt 3 Filesystem-Fix + Dashboard
⚠ Größtes Risk zuerst · Crash-Bug bleibt länger offen · Schnellerer "Wow"-Moment
C

Parallel — zwei Fronten gleichzeitig

Technische Fixes (Store + Filesystem) laufen als eigener Strang, neue Features (Workspace + Editor) als zweiter Strang.

Strang 1: Store-Refactor → Filesystem → Dashboard
Strang 2: Workspace-Section → Editor-Upgrade → Backlinks
Beide Stränge laufen unabhängig, werden am Ende zusammengeführt
⚠ Komplex · Mehr Koordination nötig · Am schnellsten fertig (wenn Ressourcen vorhanden)

Meine Empfehlung: A — solide Grundlage, kein Rework
